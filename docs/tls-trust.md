# Доверие клиента к серверу (WSS, серьёзные данные)

## Зачем это нужно

- **API key** говорит серверу «кто ты», но **не доказывает**, что ты говоришь с настоящим сервером. Без проверки TLS злоумышленник в сети может подставить свой узел и получать аудио/текст.
- **WSS + проверка сертификата** даёт шифрование канала и привязку к **идентичности сервера** (имя + ключ из доверенного источника).

## Уровни (от продакшена к деву)

### 1. Публичный CA (лучший вариант для «как в проде»)

- Сервер в интернете или внутри сети с **DNS-именем** (например `transcribe.company.internal`).
- На этом имени выпускаешь сертификат (**Let's Encrypt**, корпоративный PKI с публично доверенным CA и т.д.).
- В `client-config.json`: `"server": "wss://transcribe.company.internal:9745"`.
- **Не указывай** `tls_ca_path` и **не включай** `tls_insecure` — клиент использует системное хранилище доверенных корней (Windows).

Ограничение: имя в URL должно совпадать с **SAN** в сертификате (и не обмануть проверку hostname).

### 2. Свой корневой CA (типично для закрытого контура)

- Поднимаешь **внутренний CA**, выпускаешь для сервера **листовой** сертификат.
- Файл **`ca.pem`** (только публичный корень CA) раздаёшь клиентам **вне канала** (MDM, образ, ручная установка в доверенные — как принято у вас).
- В конфиге клиента: `"tls_ca_path": "C:\\path\\to\\company-ca.pem"` (или относительный путь от каталога запуска).
- Снова: **без** `tls_insecure`.

Плюс: ротация серверного сертификата без перевыда софта на клиентах (пока тот же CA).

### 3. Self-signed «как сейчас» (`tools/gen-dev-certs`)

- Один файл `server.pem` — это и сертификат сервера, и «свой корень».
- Клиенту: `"server": "wss://localhost:9745"` (или IP **только если** он есть в SAN сертификата).
- `"tls_ca_path": "путь/к/server.pem"` — доверяешь **именно этому** ключу, без `tls_insecure`.

`tls_insecure: true` оставь только для локальной отладки; для речевых данных в сети это не использовать.

## Что уже есть в клиенте

| Поле | Назначение |
|------|------------|
| `server` | `wss://...` для TLS |
| `tls_ca_path` | PEM своего CA или self-signed `server.pem` — **проверка сервера без отключения TLS** |
| `tls_insecure` | принять любой сертификат (**только dev**) |
| `api_key` | авторизация на сервере после установления доверенного TLS |

## Жёсткий pinning (опционально, на будущее)

Можно добавить в клиент проверку **отпечатка** (SPKI / SHA-256 сертификата) — сильнее против поддельных CA, но сложнее ротация. Сейчас достаточно CA path + нормальный выпуск сертификатов.

## Логи: как убедиться, что канал шифрованный

**Сервер** (после успешного TLS-handshake), уровень `info`:

- `transport: WSS, TLS TLSv1_3, cipher TLS_...` — зашифрованное соединение, видны версия и набор шифров.
- `transport: plain WS, TLS disabled` — без TLS (только `ws://`).

Дополнительно, уровень `debug` и target `localvox::tls`:

```bash
RUST_LOG=info,localvox::tls=debug cargo run -p live-transcribe-server-reliable -- ...
```

**Клиент:** при каждом успешном подключении в панели **Debug** в TUI пишется строка с транспортом (WSS/WS), проверкой сертификата, размером peer cert (для WSS), **`client_id`**, и напоминанием, что версию TLS/cipher смотреть в логе сервера.

## `client_id` и API key

Поле **`client_id`** в `client-config.json` нужно серверу для привязки переподключений к сессии. Если **`client_id` пустой или отсутствует**, а **API key** задан, клиент сам подставит стабильный id вида `k-` + 16 hex от SHA-256 ключа. Подробность и случай **общего ключа на два клиента** — в [client-reliable/README.md](../client-reliable/README.md) (раздел «client_id и API key»).

## Чеклист «серьёзные данные»

1. Только **WSS**, не WS, через границу доверия.
2. **Выключить** `tls_insecure` в бою.
3. Либо **публичный/корпоративный доверенный** сертификат, либо **`tls_ca_path`** на ваш CA / dev `server.pem`.
4. **API key** — отдельно, после того как TLS уже проверен.
5. SAN сертификата = тот **host**, который в `server` URL.
